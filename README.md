# protocore

> Signed binary releases for **protocore** — the [Monolythium](https://monolythium.com) node binary. Source stays private until mainnet (see [why](#why-source-is-closed-for-now)); binaries are published here so testnet operators and ecosystem developers can run real nodes today.

**Status:** preview · no releases yet · **License:** see [`LICENSE.md`](./LICENSE.md) (proprietary preview EULA — converts to BSL-1.1 at mainnet)

---

## What this repo is

A release-only repository. There is no source code here — only:

- **Binary artifacts** attached to GitHub Releases per tag (`protocore-<version>-<platform>.tar.gz`).
- **SHA-256 checksums** (`.sha256`) for every artifact.
- **Cosign signatures** (`.sig` + `.pem`) — keyless via GitHub OIDC, verifiable against the workflow identity that produced the build.
- **SBOMs** (`.spdx.json`) generated by [`syft`](https://github.com/anchore/syft).
- **Release metadata JSON** identifying the corresponding `mono-core` source commit, the Talos version of any matching Monarch OS image, and the SDK version exporting the matching RPC shape.
- A top-level [`manifest.json`](./manifest.json) index that lists every release on every channel — for tooling that needs to discover the latest compatible build without scraping the Releases page.

The actual source for `protocore` lives in the private repository `monolythium/mono-core` (see [Why source is closed for now](#why-source-is-closed-for-now)).

## Channels

Releases are organized into channels via tag prefix and the `channels` field in `manifest.json`:

| Channel | Tag prefix | Status |
|---|---|---|
| `testnet` | `v*-testnet` | First release pending |
| `mainnet` | `v*-mainnet` | Reserved — no mainnet until activation milestones land |

Each channel is independent: a testnet `v0.2.3-testnet` release can ship without a corresponding mainnet build, and vice versa. Compatibility metadata in each release identifies which Talos version, Monarch OS image tag, and SDK version it pairs with.

## Verification

Every release ships with everything you need to verify the binary came from this repo's build pipeline.

### SHA-256

```bash
sha256sum -c protocore-v0.0.1-testnet-x86_64-linux.tar.gz.sha256
```

### Cosign keyless

```bash
cosign verify-blob \
  --certificate protocore-v0.0.1-testnet-x86_64-linux.tar.gz.pem \
  --signature   protocore-v0.0.1-testnet-x86_64-linux.tar.gz.sig \
  --certificate-identity-regexp 'https://github.com/monolythium/protocore/.github/workflows/.+@refs/tags/v.+' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  protocore-v0.0.1-testnet-x86_64-linux.tar.gz
```

The certificate identity proves the artifact was produced by a workflow run on this repo, at a specific tag, under GitHub's OIDC issuer.

### SBOM

```bash
# Inspect the SPDX SBOM
cat protocore-v0.0.1-testnet-x86_64-linux.tar.gz.spdx.json | jq '.packages | length'
```

## How to use a binary

For Monarch OS operators:

1. Download the release artifact for your channel + platform.
2. Verify SHA-256 + cosign signature (above).
3. Provide the binary to Monarch OS via `PROTOCORE_BINARY=/path/to/protocore` when building `monarch-os-talos` — see that repo's [README](https://github.com/monolythium/monarch-os-talos) for the build command.
4. Provide the SHA-256 to Monarch Desktop's release-digest attestation flow (`PROTOCORE_EXPECTED_DIGEST` in the Talos extension service config) so the operator GUI can compare it against the running node's `lyth_runtimeProvenance` reply and fail closed on mismatch.

For developers running a bare-Linux operator node (testnet only):

1. Download + verify as above.
2. Run the binary against the testnet chain config (compatibility metadata in the release JSON identifies which chain ID + genesis the build was compiled for).

## Why source is closed for now

A short answer: competitive timing.

A longer answer:

Monolythium is in the pre-mainnet window where the protocol economics, the operator-cluster model, and the chain's distinguishing technical choices are still settling. Publishing the full source today before those decisions are locked invites copy-paste competitors to ship a copycat L1 ahead of mainnet — capturing attention and operator mind-share off the back of work that hasn't even seen production yet.

The plan, however, is **not** to keep source closed indefinitely:

- **At mainnet activation, the source for `protocore` flips public under [Business Source License 1.1](https://mariadb.com/bsl11/).** BSL-1.1 means the source is available to read, audit, fork, and run — with a four-year restriction on offering it as a managed commercial competing service. After four years it converts automatically to a permissive license.
- The binaries here are signed + reproducibly built (cosign keyless via GitHub OIDC, SBOMs published). External operators can verify that the binary they're running was produced by a workflow on this repo, at a specific tag, even without seeing the source.
- Adjacent ecosystem components are already public and audit-targetable: the SDK ([`mono-core-sdk`](https://github.com/monolythium/mono-core-sdk)), the operator OS recipe ([`monarch-os-talos`](https://github.com/monolythium/monarch-os-talos)), the operator console ([`monarch-desktop`](https://github.com/monolythium/monarch-desktop)), the consumer wallet ([`browser-wallet`](https://github.com/monolythium/browser-wallet)), and the developer toolchain ([`mono-studio`](https://github.com/monolythium/mono-studio)).

If the closed-binary stance is a hard blocker for a specific deployment (regulated environment, internal compliance review requiring source), email `licensing@monolythium.com` — we can talk about an evaluation-source NDA for the pre-mainnet window.

## Compatibility

Each release entry in [`manifest.json`](./manifest.json) carries a `compatibility` block identifying:

- The **Talos version** the corresponding `monarch-os-talos` image was built against
- The **`monarch-os-talos` tag** that bundles this binary as the `monarch-protocore` extension
- The **`@monolythium/core-sdk` version** exporting the RPC shape this binary serves
- The **`monarch-desktop` version** that has been smoke-tested against this binary

A mismatch isn't a hard fail — `protocore` aims to be reasonably compatible across adjacent versions of each — but the compatibility block tells you what the publishing pipeline actually verified.
